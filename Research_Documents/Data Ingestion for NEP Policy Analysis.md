# **Comprehensive Framework for Programmatic Ingestion of Educational and Macroeconomic Data**

## **Architectural Foundation of the Causal Inference Pipeline**

Evaluating the macroeconomic and institutional outcomes of the National Education Policy (NEP) across Karnataka and Tamil Nadu necessitates an exhaustively engineered data ingestion pipeline. Specifically, constructing a Difference-in-Differences (DiD) quasi-experimental design requires high-fidelity, multidimensional panel data spanning the pre-treatment and post-treatment epochs, strictly delineated between 2017 and 2025\. The architecture must simultaneously capture localized educational capacity metrics, macroeconomic labor force statistics, and institutional sentiment derived from regional news coverage. To execute this complex econometric requirement, the ingestion strategy relies on four primary application programming interfaces (APIs), each serving a distinct empirical or natural language processing (NLP) mandate.  
The National Data and Analytics Platform (NDAP) REST API operates as the primary programmatic source for official, standardized governmental statistics. NDAP fundamentally transforms the accessibility of India's administrative data by mapping disparate state and central datasets to a uniform geographic schema, ensuring high interoperability.1 The Kaggle API facilitates the retrieval of community-aggregated institutional datasets, specifically the National Institutional Ranking Framework (NIRF) archives and historical news corpora, bridging the gap left by the absence of a public-facing NIRF REST API.3 The Global Database of Events, Language, and Tone (GDELT) DOC 2.0 API provides highly granular, event-driven scraping of regional news media to construct a timeline of institutional friction, tracking real-time global news media in over a hundred languages.5 Finally, MediaStack and NewsAPI serve as redundant, localized text ingestion fallback mechanisms to guarantee a comprehensive textual corpus for advanced RoBERTa-based sentiment analysis, capturing the nuanced day-to-day coverage of regional publishers.6  
This exhaustive instruction manual outlines the exact programmatic strategies, endpoint configurations, metadata search routines, and parameter tuning required to synthesize this diverse data ecosystem into a unified data warehouse suitable for rigorous causal modeling. The subsequent sections detail how to programmatically search these data sources for requisite metadata, validate the presence of necessary fields, and set up the Python-based code bases to ingest the data payloads efficiently while respecting rate limits and authentication protocols.

## **Primary Macroeconomic and Educational Ingestion via NITI Aayog NDAP REST API**

The National Data and Analytics Platform (NDAP), spearheaded by NITI Aayog, is designed to democratize access to public government data by ensuring datasets are accessible, interoperable, and standardized to a common schema.1 For the DiD analysis, the NDAP API is the exclusive source for extracting both the All India Survey on Higher Education (AISHE) indicators and the Periodic Labour Force Survey (PLFS) metrics.9 The platform's commitment to interoperability—chiefly through the standardization of geographical boundaries using the Local Government Directory (LGD) codes—allows for the seamless merging of educational outcomes and labor market controls.2

### **Authentication, Swagger UI, and Cryptographic Handshakes**

Interacting with the NDAP ecosystem's programmatic interfaces requires the initialization of a dedicated developer environment. The platform employs a secure authentication mechanism governed by standard API access protocols, demanding an API key for access to its data payloads.2 Access requires generating an API developer key through the NDAP developer portal, which serves as the cryptographic token passed in the header of all subsequent HTTP GET and POST requests.  
The API infrastructure is heavily documented using the OpenAPI Specification (OAS), commonly interfaced via Swagger UI.11 The Swagger documentation provides a machine-readable blueprint of the NDAP schema, detailing exact endpoints, authentication requirements, rate limits, and JSON request/response formats.2 Developers must review the Swagger endpoints to understand the schema dependencies. To establish a connection, developers should initialize an automated session in a Python environment using the standard requests library. The script must instantiate a session object where the HTTP headers are updated to include the generated authorization token. This ensures that every subsequent query sent through the session inherits the authentication credentials, thereby preventing throttling, connection resets, or unauthorized access rejections during high-volume data loops.

### **Programmatic Metadata Discovery for AISHE and PLFS Schemas**

Before executing bulk extractions, the data pipeline must programmatically discover the precise dataset identifiers for the AISHE and PLFS modules. The NDAP API provides specific endpoints for dataset discovery and metadata exploration.14 A developer cannot simply query "education data"; they must parse the catalog to find the exact alphanumeric identifiers that the API database uses to route queries.  
By hitting the catalog search endpoints with query parameters such as query="AISHE" or query="Periodic Labour Force Survey", the API returns a deeply nested JSON payload containing the metadata catalog.10 The metadata payloads are critical because they define the exact variable taxonomy required for downstream extraction. The returned schema provides the unique identifier strings for the datasets containing the Gross Enrolment Ratio (GER), the total census of universities and colleges, the Pupil-Teacher Ratio (PTR), and the discipline-wise enrollment breakdowns.9  
To confirm the presence of the necessary fields, the pipeline must implement a recursive parsing algorithm that iterates through the columns or schema array within the metadata JSON response.2 For the AISHE data, the script must search the metadata strings for keywords like "Gross Enrolment Ratio", "Universities", "Colleges", and "Pupil-Teacher Ratio". Similarly, the PLFS metadata payload must be parsed to isolate the tables containing the Youth Unemployment Rate (specifically targeting the age cohort 15–29) and the Labor Force Participation Rate (LFPR) stratified by educational attainment, ensuring that outcomes for graduates and post-graduates are distinctly measurable.10 The validation algorithm must also confirm that the spatial granularity reaches the state level and that the temporal granularity covers the 2017 to 2025 longitudinal window required for the pre- and post-treatment analysis of the DiD methodology.

### **Payload Construction, Geographic Standardization, and Temporal Iteration**

Once the target dataset identifiers and their exact column names are isolated through the metadata discovery phase, the ingestion script must construct the extraction payload. The NDAP API allows for advanced filtering natively at the endpoint level, minimizing the amount of data transferred over the network and optimizing local memory usage.2 The two most critical parameters for this causal research are the geographic filter and the temporal filter.  
NDAP relies on Local Government Directory (LGD) state codes to enforce spatial standardization across all ministries and administrative units.2 The programmatic script must utilize a parameter mapping dictionary to translate the text inputs of the target regions into the LGD integer codes. According to the LGD standards, the state code for Jammu and Kashmir is 1, but critically for this study, the state code for Karnataka is 29 and the state code for Tamil Nadu is 33\.16 The payload must explicitly pass a JSON array to the state filter parameter containing these precise integer values (\`\`) to restrict the returned rows strictly to the required geographic treatment (Karnataka) and control (Tamil Nadu) units.  
To satisfy the temporal requirements of the DiD framework, the ingestion script must iterate through the analytical timeframe. This is best achieved by wrapping the API call inside a temporal looping function in Python that systematically modifies the year parameter in the request payload from 2017 through 2025\. During each iteration, the script executes a POST request to the dataset query endpoint, passes the LGD geographic filters and the specific year, and awaits the JSON response.  
Upon receiving the data payload, the script must utilize the Pandas library to normalize the JSON output into a structured tabular format. The date-time indices must be standardized to represent the academic or financial year accurately, and the cross-sectional data must be concatenated into a unified, multi-index longitudinal panel dataframe. This multi-index structure, utilizing state and year as the primary keys, perfectly aligns with the standard econometric inputs required for executing a Difference-in-Differences regression.

## **Micro-Institutional and Historical Text Ingestion via Kaggle API**

While the official NDAP platform provides robust, standardized macro-level statistics, evaluating the specific institutional realities of the NEP rollout requires highly localized performance data. The policy emphasizes university consolidation mandates, multidisciplinary shifts, and improvements in graduation outcomes. Because the official National Institutional Ranking Framework (NIRF) portal lacks a public-facing REST API for programmatically querying historical metrics, the Kaggle API serves as the critical ingestion vector for community-aggregated institutional data.3 Furthermore, Kaggle hosts substantial, pre-compiled historical news corpora, which are absolutely necessary for training, testing, and calibrating the RoBERTa sentiment architecture before it is deployed on live API streams.

### **Environment Initialization and Package Authentication**

Interacting with the Kaggle ecosystem requires the installation and programmatic configuration of the official kaggle or kagglehub Python package.3 The authentication mechanism deviates from standard HTTP header tokens passed in requests; instead, it relies on a local configuration file or strictly defined environment variables.18  
To configure this environment programmatically, the developer must first generate an API token from their Kaggle account dashboard, which triggers the download of a kaggle.json file containing the username and cryptographic key.18 In a Linux or macOS development environment, this file must be programmatically or manually moved to the \~/.kaggle/ directory, while on Windows environments, it must reside in C:\\Users\\\<username\>\\.kaggle\\.18 However, for robust, scalable data pipelines—such as those deployed in Docker containers or cloud-based Jupyter notebooks—the authentication protocol is best fulfilled by passing the credentials directly into the system environment variables KAGGLE\_USERNAME and KAGGLE\_KEY.18  
Once the environment variables are populated, the script instantiates the connection by importing the API class via from kaggle.api.kaggle\_api\_extended import KaggleApi, instantiating the object with api \= KaggleApi(), and executing the api.authenticate() method.18 This routine establishes a secure, rate-limited connection to the Kaggle servers, preparing the environment for bulk dataset downloads. Developers must remain cognizant of dynamic rate limiting; encountering HTTP 429 errors requires implementing exponential backoff and retry logic within the ingestion scripts.3

### **Querying and Extracting NIRF Placement and Outcome Metrics**

The data extraction pipeline requires highly specific institutional variables to gauge the success of the educational policies: absolute placement numbers, median salaries to gauge the economic quality of placements, the volume of students opting for higher studies, and the ratio of approved intake to total graduated students to calculate precise retention and dropout rates.19  
The developer must utilize the Kaggle API's search functionality or directly reference known, high-quality dataset slugs. Datasets such as nagarshivam/india-colleges-and-cities and atharvmohanjadhav/nirf-2024-college-rankings-dataset contain rich, tabular repositories of these institutional metrics.4 The metadata for these datasets confirms the presence of necessary analytical variables like fees\_ug\_inr, placement\_avg\_lpa, nirf\_rank, and highly structured sub-scores covering Teaching, Learning & Resources (TLR), Research & Professional Practices (RPC), and Graduation Outcomes (GO).4  
To extract the data, the ingestion algorithm invokes the dataset\_download\_files() function provided by the Kaggle Python API.17 The function requires the dataset slug, which is formatted consistently as \[owner\]/\[dataset-name\].17 The developer must set the path parameter to route the downloaded archives directly into the data/raw/ directory of the local project structure.17 To optimize the pipeline, the script must trigger the unzip=True parameter, which instructs the API client to automatically decompress the archived CSV files upon successful transfer, removing the need for secondary shell commands to handle zip extraction.17 Once decompressed, Pandas is used to load the CSVs, filter the rows where the state column matches Karnataka or Tamil Nadu, and isolate the columns pertaining to approved intake, placed students, and median salary.

### **Ingesting the "A Million News Headlines" Corpus for Baseline NLP Calibration**

To establish a mathematical baseline for the sentiment analysis architecture, large-scale historical text datasets are required to train the embedding layers. Kaggle hosts the highly utilized "A Million News Headlines" dataset, containing news metadata published over more than fifteen years, sourced from reputable broadcasting corporations.21 The schema for this dataset is heavily optimized for natural language processing and topic modeling, containing two primary, easily parsed columns: publish\_date and headline\_text.23  
The ingestion script must download this dataset using the exact same dataset\_download\_files(slug='therohk/million-headlines', path='data/raw', unzip=True) methodology.17 Upon extraction, the script must initialize a rigorous data cleaning function using Pandas. The publish\_date column natively arrives as a concatenated integer or string in the YYYYMMDD format (e.g., 20030219).23 The script must cast this column into a standard datetime object using the vectorized function pd.to\_datetime(df\['publish\_date'\], format='%Y%m%d') to ensure compatibility with the temporal alignment required by the Difference-in-Differences setup.23 Following the datetime conversion, strict filtering logic is applied to isolate records falling exclusively within the 2017 to 2025 analytical window. This cleaned, temporally bounded corpus is then utilized to calibrate the tokenizers for the RoBERTa model, ensuring the NLP architecture understands the baseline distribution of journalistic terminology before analyzing the highly specific NEP discourse.

## **Event-Driven Narrative Scraping via GDELT DOC 2.0 API**

While Kaggle provides broad historical baselines, understanding the real-time, event-driven friction points of the NEP implementation—such as protests over the University Grants Commission's Indian Knowledge Systems (UGC IKS) mandates or the administrative failures regarding the Karnataka Examinations Authority (KEA) seat-blocking crisis—requires highly precise, localized web scraping. The Global Database of Events, Language, and Tone (GDELT) Project monitors global news media in over a hundred languages, generating a real-time, massive network diagram of human society.5 The GDELT DOC 2.0 API provides a direct, programmatic pathway to query this vast, continuously updating database.25

### **Endpoint Configuration and URL Parameterization Mechanics**

The developer must configure the NLP extraction pipeline to hit the dedicated DOC 2.0 endpoint: https://api.gdeltproject.org/api/v2/doc/doc.25 Unlike standard REST APIs that rely on deep JSON bodies for querying and complex header structures, the GDELT DOC 2.0 API is driven entirely by URL parameters, making it highly efficient for rapid, iterative text extraction and simple to implement using standard HTTP GET requests.25  
To pipe the results directly into a structured database or a Pandas dataframe for analytical processing, the FORMAT parameter within the URL must be explicitly set to json or csv.25 The default response format of the API is an HTML browser-based visualization, which is completely incompatible with automated NLP ingestion pipelines.25 Furthermore, the MODE parameter defines the specific structure and depth of the data returned; setting MODE=ArtList forces the API to return a structured list of individual articles matching the query, containing the source URL, the title of the article, and GDELT's pre-calculated sentiment metrics.25  
The temporal scope of the DOC 2.0 API is remarkably deep, allowing for full-text searches of news coverage dating back to January 1, 2017\.25 This seamlessly covers the entire pre-treatment and post-treatment window required for evaluating the NEP. However, the default behavior of the API searches only the most recent 24 hours to 3 months of coverage depending on the query load.25 Therefore, the ingestion script must programmatically override this default by defining the temporal window using the STARTDATETIME and ENDDATETIME parameters. These parameters strictly accept precise timestamps formatted as YYYYMMDDHHMMSS.25 To extract the massive 2017–2025 window without triggering server-side timeout errors or exceeding query size limits, the Python algorithm should dynamically loop through the timeline in distinct monthly or quarterly increments, updating the STARTDATETIME and ENDDATETIME strings, executing the GET request, and aggregating the JSON responses in memory before writing the final corpus to disk.

### **Constructing Boolean Logic and Advanced Spatial Filters**

The core functionality and precision of the GDELT pipeline lie in the construction of the QUERY parameter. This parameter supports complex Boolean logic, exact phrase matching, domain filtering, and advanced thematic operators.25  
To isolate the specific institutional narratives surrounding the educational reforms, the script must construct a compound boolean search query using URL-encoded strings. The topic vectors must capture the policy itself, utilizing exact phrase matching where necessary: ("NEP" OR "National Education Policy" OR "Higher Education").  
Crucially for the Difference-in-Differences design, the query must be constrained exclusively to the target treatment and control geographies. GDELT supports granular spatial filtering via the location: operator.29 A rudimentary textual filter can be applied by appending AND (location:"Karnataka" OR location:"Tamil Nadu") to the query string. However, relying solely on text matching for geography can result in significant ambiguity, especially when dealing with smaller municipalities that share names globally. For precise, unambiguous geocoding, the script should leverage the First Order Administrative Division (ADM1) codes provided by the Global Administrative Unit Layers (GAUL) standard integrated into GDELT 2.0.30  
In the GDELT geographical taxonomy, India as a nation is coded as IN.33 To filter at the state level, Karnataka is specifically represented by the ADM1 code IN19, and Tamil Nadu is represented by the ADM1 code IN25.33 Implementing a query utilizing these specific geographic entity codes ensures that the linguistic data returned strictly pertains to events occurring within the designated treatment and control boundaries, drastically reducing the noise of national-level policy discourse.

### **Extracting Tone and Preparing the Corpus for RoBERTa Sentiment Classification**

A highly unique and valuable feature of the GDELT DOC API is its native sentiment evaluation engine. When querying in ArtList mode with the JSON format specified, the payload includes a pre-calculated Tone metric for every returned article.25 This tone score represents the average sentiment of the document, typically ranging mathematically from \-100 (indicating extreme negativity or conflict) to \+100 (indicating extreme positivity or cooperation), though the vast majority of standard news media clusters tightly between \-10 and \+10.25  
While the ultimate goal of the data pipeline is to deploy a state-of-the-art RoBERTa transformer model for deep contextual sentiment analysis, the GDELT native tone score serves as a critical baseline and programmatic filtering mechanism. The ingestion script can extract this tone value during the parsing phase and use it to flag articles exhibiting severe negativity. These flagged articles are often highly indicative of the specific implementation failures and administrative friction points hypothesized in the DiD model, allowing researchers to prioritize texts that discuss protests, seat-blocking, or administrative gridlock.  
Furthermore, if the developer utilizes the GDELT Context 2.0 API—a functional sibling to the DOC 2.0 API—the system can return specific textual snippets showing the exact context surrounding the matching keywords.26 This enforces a strict relevancy requirement where all target keywords must appear within the exact same sentence.26 This contextual snippet extraction is invaluable for feeding exact friction-point sentences into the RoBERTa architecture. Analyzing single sentences rather than full documents dramatically reduces the noise inherent in full-article NLP processing, preventing the model from misinterpreting the overall sentiment of a long, multifaceted journalistic piece.

## **Redundant Regional Text Scraping via MediaStack and NewsAPI**

The inherent volatility of global event databases means that highly localized, vernacular-adjacent coverage of niche academic policies might occasionally slip through the broad GDELT indexing protocols. Furthermore, global aggregators sometimes struggle to maintain persistent links to regional publishers over an eight-year span. Therefore, the data infrastructure requires a robust, redundant layer of text ingestion using standard commercial news APIs: MediaStack and NewsAPI.6 These APIs act as fail-safes, capable of querying specific regional news publishers (such as the Deccan Herald in Karnataka or The Hindu in Tamil Nadu) to capture the granular, day-to-day friction of the NEP implementation.34

### **MediaStack Integration, Constraint Management, and Targeted Batch Processing**

MediaStack operates as a highly scalable RESTful API delivering lightweight JSON payloads specifically optimized for news aggregation.35 To integrate MediaStack into the Python pipeline, the developer must register for a free-tier API access key, which acts as the authentication token. Unlike NDAP which requires headers, MediaStack authenticates by appending this key directly to the base URL via the access\_key parameter.34  
The base endpoint for extracting articles is http://api.mediastack.com/v1/news.38 The API allows for targeted geographic filtering by passing the ISO-2 country code parameter. For this project, setting country=in restricts the output specifically to Indian news sources, eliminating international noise.34 The developer can then append the keywords parameter to search for terms highly specific to the implementation failures identified by previous analyses, such as "seat matrix", "UGC IKS", or "NEP protests".  
However, managing rate limits is the primary challenge when utilizing this API. The free tier of MediaStack is heavily constrained to 500 requests per month.39 Given this tight limitation, the ingestion script cannot be designed to stream continuous data or execute massive daily loops. Instead, it must be architected as a highly targeted batch processor. The Python code should calculate the exact dates of known policy announcements, major implementation deadlines, or significant protests, and selectively fire queries targeted specifically around those narrow temporal windows. This intelligent querying strategy conserves request volume while maximizing the extraction of relevant institutional friction data.

### **NewsAPI Endpoint Configuration, Pagination Logic, and Domain Filtering**

NewsAPI provides an alternative, structurally similar REST interface for text extraction.41 The service offers two primary endpoints: /v2/top-headlines for live breaking news, and /v2/everything for deep historical searches.43 Because the econometric analysis requires data spanning back to 2017, the /v2/everything endpoint is the mandatory target for the ingestion script.44  
Authentication is handled securely by passing the API token via the apiKey URL parameter or embedded within the HTTP headers.44 Like MediaStack, the query parameters dictate the extraction logic. The q parameter handles boolean keyword searches, and the domains parameter can be utilized to restrict scraping exclusively to known regional publishers relevant to Karnataka and Tamil Nadu (e.g., domains=thehindu.com,deccanherald.com). This domain-level filtering severely narrows the focus and eliminates the dilution caused by national-level broadcasters.42  
However, the developer must carefully navigate NewsAPI's tier limitations and payload structures. The free Developer plan restricts the search horizon to articles published within the last month and caps usage at 100 requests per day.43 Therefore, executing a longitudinal extraction from 2017 to 2025 programmatically requires upgrading to an Academic, Business, or Advanced tier key, which permits searching articles up to five years old or beyond depending on the bespoke enterprise agreement.43  
Assuming proper credentialing is secured, the script must implement strict pagination logic. Because NewsAPI returns a maximum number of articles per JSON payload (typically 100), the code must parse the totalResults integer provided in the initial JSON response. The script then calculates the required number of pages by dividing totalResults by the page size. Finally, it must iteratively execute a while loop, incrementing the page parameter in the URL until the entire regional corpus matching the query is successfully downloaded and concatenated into the local database.

This enforced uniformity across all text sources guarantees that the RoBERTa model consumes a completely standardized textual format, regardless of whether the text originated from GDELT, Kaggle, MediaStack, or NewsAPI.

## **Synthesizing the Unified Data Warehouse for Difference-in-Differences Modeling**

The exhaustive programmatic ingestion outlined above results in the construction of two distinct but deeply interrelated data warehouses: an empirical matrix of macroeconomic and educational statistics derived from NDAP and Kaggle, and a vast, temporally indexed corpus of institutional narratives aggregated from GDELT, MediaStack, and NewsAPI. The ultimate success of the causal inference architecture depends entirely on the rigorous programmatic harmonization of these disparate structures into a singular panel dataframe.

### **Harmonizing Spatial-Temporal Indices Across Disparate APIs**

The theoretical core of a Difference-in-Differences model relies on observing the parallel trends of a treatment group (Karnataka, representing early NEP adoption) and a control group (Tamil Nadu, representing resistance or modification to the NEP) over time. The objective is to measure the divergence of outcomes in the post-treatment period, isolating the causal effect of the policy intervention. The structural econometric equation typically follows the form:  
$Y\_{ist} \= \\alpha \+ \\beta (Treatment\_s \\times Post\_t) \+ \\gamma X\_{ist} \+ \\epsilon\_{ist}$  
The NDAP API pipeline directly populates the critical dependent variables ($Y\_{ist}$). The AISHE dataset extracts supply the structural indicators of the higher education system: the Gross Enrolment Ratio (GER), changes in the Pupil-Teacher Ratio (PTR), and the absolute count of operational institutions. The variation in the total number of universities over time directly tracks the success or failure of the NEP’s aggressive consolidation mandates, which aimed to merge smaller colleges into large, multidisciplinary universities. Concurrently, the Kaggle API pipeline supplements this empirical foundation with micro-level institutional outcomes. By merging the Kaggle NIRF datasets with the NDAP spatial taxonomy, the pipeline injects crucial variables regarding dropout rates (approved intake vs. total graduated) and the median salary of placed students—metrics that dictate whether the educational reforms are yielding tangible, high-quality human capital improvements.4  
However, educational outcomes do not occur in an economic vacuum; they are heavily confounded by broader regional and macroeconomic realities. This dictates the absolute necessity of the PLFS dataset extraction from the NDAP portal.10 By injecting the Youth Unemployment Rate and the Labor Force Participation Rate (LFPR) of graduates into the design as dynamic control covariates ($X\_{ist}$), the econometrician stabilizes the model against exogenous macroeconomic shocks. If placement rates decline sharply in Karnataka post-2020, the inclusion of the PLFS data allows the model to mathematically discern whether the decline is a failure of the NEP curriculum modifications, or merely a symptom of a broader regional economic downturn affecting all sectors simultaneously. To achieve this, the Python script must execute a series of pd.merge() operations, joining the AISHE, NIRF, and PLFS datasets strictly on the LGD state codes and the standardized academic year integers.

### **Bridging NLP Sentiment Outcomes with Econometric Panel Data**

The integration of the NLP pipeline elevates the evaluation beyond standard econometrics, allowing researchers to quantify institutional friction. The GDELT DOC API, MediaStack, and NewsAPI extractions provide millions of data points representing the daily, localized friction of policy implementation, capturing the qualitative reactions of students, faculty, and administrators.5 The ingestion scripts align these textual records along the identical temporal axis as the annual NDAP and Kaggle data by normalizing all publication timestamps into corresponding academic years.  
Once ingested and cleaned, the text strings (headlines and descriptions) are tokenized and processed through a fine-tuned RoBERTa transformer architecture. RoBERTa evaluates the deep semantic context of phrases like "KEA seat-blocking crisis," "UGC multidisciplinary mandates," or "faculty protests," assigning a high-resolution sentiment probability vector to each article. Unlike older lexical models, RoBERTa understands the bidirectional context of words, recognizing that "blocking" in the context of university seats is highly negative, whereas "blocking" in a cybersecurity context might be positive.  
These granular sentiment scores are then aggregated temporally. The Python script calculates the average weekly, monthly, or annual institutional hostility index for Karnataka versus Tamil Nadu. This derived sentiment metric is subsequently merged into the main econometric panel dataframe as a continuous variable. Functionally, this allows the sentiment index to act as an advanced, high-frequency leading indicator within the regression model. The data scientist can mathematically model whether a spike in negative institutional sentiment (captured by GDELT and evaluated by RoBERTa) in year $t$ successfully predicts a statistically significant collapse in student retention, GER, or placement quality (captured by the NDAP and Kaggle datasets) in year $t+1$.  
By rigorously adhering to these comprehensive API extraction protocols—leveraging NDAP's standardized LGD schemas, Kaggle's historical archives and community metrics, GDELT's event-driven geographic mapping, and commercial fallback scraping via MediaStack and NewsAPI—the resulting data infrastructure achieves the scale, granularity, and cross-disciplinary scope necessary. This architecture allows the expert data scientist to definitively evaluate the complex institutional realities, friction points, and ultimate causal impacts of the National Education Policy.
